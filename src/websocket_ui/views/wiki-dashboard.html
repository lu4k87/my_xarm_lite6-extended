Dashboard Architektur & Datenfluss

Hier wird im Detail erklärt, wie das Dashboard im Hintergrund arbeitet, welche Dateien beteiligt sind und wie die Live-Daten aus ROS 2 extrahiert und kategorisiert werden.

1. Backend (Datenerfassung & ROS 2 Bridge)

workspace_analyzer.py

Kernstück: Hybrid-Scanner & Daten-Aggregator
Statische Analyse: Durchsucht den ~/dev_ws/src Ordner nach Projekt-Dateien (Python/C++), analysiert package.xml auf Abhängigkeiten und zerlegt rekursiv Launch-Files, um Start-Beziehungen (launched_by) zu verstehen.
Live-Metadaten via CLI: Liest im Hintergrund asynchron per Subprozess echte ros2 node info Befehle aus. Dies garantiert, dass zu aktiven Nodes die korrekten Topics, Services und Clients gemeldet werden, anstatt sich auf fehleranfälliges Regex zu verlassen.
Die is_workspace Flag: Berechnet für jeden gefundenen Node dynamisch, ob dieser aus eurem eigenen Workspace stammt (z.B. indem geprüft wird, ob die Node Exe dem lokalen Workspace gehört oder in einem lokalen Launch-Script aufgerufen wird).

rosbridge_server

Der Übersetzer
Offizielles ROS 2 Paket. Öffnet einen WebSocket-Port (9090).
Konvertiert rohe ROS 2 Messages in standardisiertes JSON für den Web-Browser und leitet umgekehrt UI-Befehle ins ROS Netzwerk weiter.

2. Node Explorer Logik (Sidebar)

Das Dashboard trennt alle erkannten ROS 2 Nodes intelligent in drei Hauptkategorien. Die Zuordnung erfolgt durch das JavaScript-Frontend (dashboard_script.js), welches die is_workspace Flag vom Backend auswertet.

Workspace Nodes

Eigener, aktiver Code.
Diese Nodes laufen gerade im System (grüner Status-Puls). Das Backend hat validiert, dass ihr Ursprung im lokalen dev_ws/src Ordner liegt – sei es als kompiliertes C++ Executable, statisches Python Script oder durch ein lokales Launch File parametrisiert.

System Nodes

Global installierte Programme.
Diese Nodes laufen aktiv mit, haben aber keine direkte physikalische Entsprechung im gecachten Workspace. Beispiele: rosbridge_websocket, MoveIt Controller (/servo_server), global installierte ROS-Packages.

Smart-Clustering:

Das Frontend gruppiert unübersichtliche Systemkomponenten automatisch. So werden z.B. alle Daemons in einem Ordner System Daemons (CLI) versteckt und Marker-Visualisierungen unter RViz 2 geordnet.

Inaktive Dateien

Der Code-Scanner hat diese .py oder .cpp Datei mit ROS-Code im Workspace gefunden, aber zum jetzigen Zeitpunkt ist kein laufender Node damit assoziiert. Sie eignen sich zur Source-Code-Ansicht.

3. Live-Datenfluss (Wie kommen Sub/Pub Topics ins UI?)

Der dynamische Topic Tracker

Schritt 1: UI Trigger
Klickt der Benutzer auf einen Node, extrahiert die JS-Logik dessen Publisher und Subscriber.
JS sendet eine Anforderung per WebSocket an /ui/request_topic_activity inklusive aller benötigten Topics.
Schritt 2: Backend Bindings
Der workspace_analyzer löst die Topic-Strings dynamisch über get_message(type) auf.
Es werden temporäre ROS-Subscriptions erstellt, welche bei Message-Eingang einen Zähler hochsetzen und die letzte Payload cachen.
Schritt 3: Frequenz-Kalkulation
Ein 500ms Timer prozessiert die angehäuften Zähler, vergleicht sie mit der Delta-Zeit und errechnet die Frequenz (Hz) der Messages.
Schritt 4: Render Response
Das Ergebnis-Tuple (Hz, ActiveState, stringified(MsgPayload)) wird JSON-gepackt auf /dashboard/topic_activity gepublished.
Das UI liest das Ergebnis, aktiviert die Pulse-Animationen (grün leuchtender Punkt) an den Flow-Linien und updatet den DOM-Text.

Frontend (UI Layout)

dashboard_index.html
Definiert das DOM-Grundgerüst (Container für Flow, Tab-Leiste, Explorer). Enthält die roslibjs Imports um nativ via JS mit ROS sprechen zu können.
dashboard_style.css
Ein hochentwickeltes Flex/Grid-System mit CSS Variables. Bestimmt die Glow-Effekte, abgerundeten Rahmen und Scrollbars (Dark Theme).
views/
Enthält HTML-Fragmente für die Tabs (zB das Wiki, Netzwerk-Ansicht). Werden zur Laufzeit vom VanillaJS in den Content-Bereich geschoben (Single Page Application Prinzip).
